## Job Sheets
Job sheets are required as part of the order process followed by Paddocks, see diagram above for more information on the process.

This action is aimed at orders with **status** "**processing**" and have an invoice.

### Usage
- navigate to the sales order grid
- select all the orders you want to print job sheets for
- from the mass actions, select "print job sheets"
- this will open a new tab that will have the job sheets for any valid orders.
- it will also automatically open the print dialog (you will notice that when printed each job sheet will start on a new page, though this isn't visible from the html displayed in the web page)
- Once the new tab is closed, the original page (where you selected the orders) will be refreshed
- All orders that were at **status** "**processing**" and have an invoice will have been moved to **status** "**processing job sheet printed**" (this is so paddocks can keep track of which orders have had their job sheet printed)
- Any orders that were invalid will have a corresponding error message displayed on sales order page after refresh
- If an order is valid (has an invoice), but not in the correct status, will still have a job sheet printed, but will not have its status changed

### Prerendering
In previous implementations of this functionality paddocks experienced timeouts, due to the number of orders that needed to be rendered. (Sometimes selecting 100's at once)
To try and combat this there is a cronjob that runs (`*/5 * * * *`) in the backround and attempts to render the jobsheet for an order (same logic as above), though it **will not** change an order's status.
Job sheets are stored in: `var/job_sheets/order_id-template-version.html`
**N.B** the template version is a date, but this **doesn't** signify the date the file was created

### Clean up
As the job sheets will be stored in `var/job_sheets` this directory is cleaned up nightly (`15 1 * * *`), deleting any files that haven't been access in more than 30 days.

### Job Sheet Template
To allow changes to be made to the job sheet templates:
- without requiring a downtime release
- be used for orders that have already got a job sheet rendered
  There is a const `Zero1\JobSheets\Service\Renderer::HASH_VERSION` which can be updated, to force the new template to be used for all orders going forward regardless of if they already have one generated.

### Sending Order Data To Selling Point
Order data is automatically sent to Selling Point when all items have been invoiced.
Once an order has sent (or failed to send) to Selling Point a comment on the order will be added:
**Success**:
```
Sent order data to Selling Point. (initiator: Magento: Order Invoiced)
```
**Fail**:
```
Failed To Send order data to Selling Point. (initiator: Magento: Order Invoiced)
```
> In both cases the `initiator` may change if a developer has been involved.

## PCI Scans

To create a PCI scan report, login here:
https://pci.qualys.com/merchant/login.php

(ask Callum for credentials he will share them securly)

Once in, on the sidebar navigate to 'Network' -> 'New Scan'.

Call the scan for example: 'Monthly scan - November 2021'
Leave everything else default, make sure target IPs is 'All IPs', and Launch is 'Launch Now'.

## CCL Integration
Paddocks has a module called: `Zero1_CCLIntegration`. It depends on `Zero1_BulkOrderShip`

`Zero1_CCLIntegration` fetches files from a SFTP endpoint and processes them, once processed it moves them to a processed endpoint.
